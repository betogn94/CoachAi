01 VISIÓN GENERAL Y OBJETIVOS

CoachAI es una aplicación web que pone un entrenador personal de bolsillo en manos de cualquier persona que quiera transformar su físico de forma sostenible, y que al mismo tiempo le da a los coaches profesionales una herramienta para escalar su trabajo sin perder la cercanía con cada alumno.

OBJETIVO PARA EL USUARIO

El usuario llega buscando una respuesta concreta a una pregunta simple: ¿qué hago hoy? CoachAI elimina la fricción de planificar comidas, decidir entrenamientos y registrar el progreso. La aplicación genera planes semanales de rutina y dieta adaptados a sus datos, su objetivo y su nivel, y mantiene un diálogo constante a través del chat para ajustar lo necesario.

  • Claridad diaria — sabe qué comer y qué entrenar cada día, sin tener que pensarlo.
  • Adherencia visible — su progreso semanal se ve reflejado en estadísticas, gráficos y comparativa visual con fotos.
  • Feedback honesto — al cerrar cada semana recibe un análisis personalizado, motivador pero basado en datos reales.
  • Conversación, no formularios — todo cambio de plan se pide hablando con CoachAI, como con un entrenador humano.

OBJETIVO PARA EL COACH

El coach administra una cartera de alumnos sin perder el detalle. El panel entrenador resume el estado de adherencia, separa a los alumnos por color de prioridad (comprometidos, adherencia media, inactivos) y permite ver el detalle completo de cada uno: planes activos, registros diarios, fotos de progreso y los análisis semanales generados.

  • Vista de cartera — cuántos alumnos están comprometidos, cuántos necesitan un empujón, cuántos están inactivos.
  • Diagnóstico individual — al entrar al perfil de un alumno, el coach ve su historial completo de semanas y las decisiones que la IA tomó por él.
  • Trazabilidad de planes — cada rutina y dieta queda registrada con su estado (ACTIVA, CERRADA, HISTÓRICA) y la conversación que la originó.
  • Escala sin perder cercanía — el coach interviene solo donde es necesario, mientras la IA mantiene el día a día.

Aparece en el logo, en el botón INGRESAR, en el botón de generar análisis y en cada elemento donde queremos comunicar "esto es lo importante".

04 ARQUITECTURA DE LA APLICACIÓN

CoachAI está construida como una single-page-app que se mueve entre siete pantallas-wrapper mutuamente excluyentes. No hay router tradicional con URLs por sección; cada pantalla se muestra y se oculta vía JavaScript, lo que da transiciones instantáneas sin recarga.

FLUJO PRINCIPAL

LANDING → ONBOARDING (5 pasos) → HOME / CHAT
                     ↓
MI DÍA   PROGRESO   PERFIL

/admin → PANEL ENTRENADOR

PERSISTENCIA

Los datos viven en Supabase (Postgres + Storage). Las tablas principales son:

  • usuarios — perfil, edad, sexo, peso, altura, objetivo, días de entrenamiento, equipamiento.
  • planes_semanales — rutinas y dietas generadas, con estado activo/cerrado.
  • diario_alimenticio — registros diarios de comidas, entrenos y peso.
  • cierres_semanales — análisis de IA, foto de la semana y stats agregadas.
  • fotos_progreso — fotos clasificadas por tipo (inicio, progreso, semana).
  • historial_chat — toda la conversación con CoachAI, fuente de verdad de qué se pidió.

INTELIGENCIA

El motor conversacional usa el API de Anthropic Claude. Cada mensaje del usuario se envía con un system prompt que incluye sus datos, sus planes activos y el historial reciente. La IA puede generar planes nuevos, responder dudas, hacer análisis semanales y comparar fotos antes/después.

UNA SOLA PUERTA

El landing es la única forma de entrar. No hay registro y login separados: con el nombre y el email, el sistema reconoce si el usuario ya existe (y lo manda directo al home) o si es nuevo (y lo manda al onboarding).

POR QUÉ EL DISEÑO ES MINIMAL

El producto pelea contra la fricción. Cuanto menos haya que pensar para entrar, mayor la conversión. Por eso el formulario tiene solo dos campos, el botón ocupa todo el ancho del card y el fondo es una foto de gimnasio con un overlay de marca, que recuerda inmediatamente para qué se descargó la app.

ACCESO ADMINISTRADOR

Debajo del card, un link discreto lleva al panel del coach. Está intencionalmente sin protagonismo: el 99% de los usuarios nunca lo va a tocar, pero el coach lo encuentra al primer barrido.

LOS 5 PASOS

  • 01 · DATOS BÁSICOS — nombre, email, edad, sexo, altura, peso. Lo mínimo demográfico.
  • 02 · COMPOSICIÓN ACTUAL — peso actual, cintura, nivel de actividad fuera del entrenamiento.

POR QUÉ EL HEADER QUEDA DESHABILITADO

Durante el onboarding, el logo y el menú hamburguesa se desactivan. Esto evita que un toque accidental saque al usuario del flujo y lo lleve a una pantalla vacía. Es una decisión chica pero corrige una de las primeras frustraciones detectadas en QA real.

FONDO COHERENTE

El onboarding usa el mismo fondo abstracto de gimnasio que el resto de las secciones autenticadas. Esto crea continuidad: el usuario "ya está adentro" desde el primer paso.

06 ONBOARDING · DETALLES

QUÉ HACE LA IA CON ESTOS DATOS

Apenas el usuario termina el paso 5, los datos quedan persistidos en la tabla usuarios y la aplicación abre el chat con un mensaje de bienvenida. A partir de ese punto, cuando el usuario pide su primera rutina o dieta, el system prompt enviado a Claude incluye:

  • Edad, sexo, altura y peso actual para calcular requerimientos calóricos y proteicos.
  • Objetivo declarado para definir el déficit, superávit o mantenimiento.
  • Días disponibles para distribuir los entrenamientos.
  • Equipamiento para no recetar ejercicios imposibles.
  • Alergias e intolerancias para excluir ingredientes.

VALIDACIONES Y TRAMPAS COMUNES

  • Edad se restringe entre 14 y 80 años — la app no atiende a menores ni propone planes que requieran supervisión médica especializada.
  • Peso y altura aceptan solo valores plausibles para evitar planes generados a partir de errores de tipeo.
  • El paso 5 es opcional — el usuario que no quiera subir foto puede saltearlo. Si nunca sube una baseline, la comparativa visual semanal queda oculta hasta que lo haga.

FONDO Y DESARROLLO VISUAL

Cada sección del onboarding muestra el número grande en gradiente violeta-cyan en una esquina, sin iconos al lado. Probamos versiones con iconos y resultó saturado: el número solo, en Bebas Neue, con suficiente espacio, lee como "Capítulo 03". Decisión deliberada después de iteración.

POSIBILIDAD DE EDITAR DESPUÉS

Todo lo que el usuario completa en el onboarding puede editarse luego desde la sección Perfil → Editar datos del perfil. La aplicación nunca obliga a "volver a empezar" — los datos se ajustan en sitio y los planes siguientes los respetan.

EL CORAZÓN DEL PRODUCTO

El chat es donde sucede todo. El usuario habla en lenguaje natural — "armame la rutina de esta semana", "me re engordé este fin de semana, qué hago el lunes", "cambiame la cena del jueves porque no tengo pollo" — y la IA responde, con la posibilidad de generar planes completos cuando hace falta.

DISTINCIÓN ENTRE PLAN ACTIVO Y CERRADO

El system prompt enseña a la IA a respetar el ciclo del plan: una rutina o dieta se considera activa mientras el usuario no haga el cierre semanal. Si pide cambios menores, se ajusta sobre la misma. Solo cuando el usuario explícitamente cierra la semana o pide un plan nuevo se genera otro.

HISTORIAL COMPLETO

Toda conversación queda guardada en historial_chat. Al volver a entrar, el usuario ve los últimos mensajes y la IA tiene acceso al contexto sin reintroducciones.

EL RITUAL SEMANAL

El cierre es el momento en que el usuario hace introspección. El modal muestra primero los números fríos (entrenó, comió, registró), después la comparativa visual (foto inicial vs foto de la semana) y finalmente entrega el análisis de la IA.

BASELINE INTELIGENTE

El slot izquierdo siempre carga la foto baseline real del usuario, nunca una foto semanal. La función pickBaselinePhoto prefiere fotos marcadas como "inicio", y si no hay, toma la más antigua que NO sea de cierre semanal — evita el bug clásico de comparar la foto de cierre contra sí misma.

ANÁLISIS HONESTO

La IA recibe los stats numéricos como fuente primaria y las dos fotos como contexto visual. Genera 3-4 párrafos en tono cercano, con foco en lo que funcionó y lo que hay que ajustar para la próxima semana. No diagnostica ni estima porcentaje de grasa corporal.

QUÉ PASA DESPUÉS

Al cerrar la semana, los planes activos quedan marcados como CERRADOS. La siguiente vez que el usuario pida una rutina o dieta, la IA genera planes nuevos para el ciclo siguiente.

SEMÁFORO DE ADHERENCIA

Las tarjetas de stats agrupan a los alumnos por color: verde (comprometidos), amarillo (adherencia media), rojo (inactivos). Tocar cada tarjeta filtra la lista a ese segmento. El coach va directo a quienes necesitan intervención.

DETALLE INDIVIDUAL

Al entrar al perfil de un alumno, el coach ve:

  • Datos demográficos y objetivo declarado.
  • Plan activo de rutina y dieta, con la conversación que los generó.
  • Historial de planes con badge ACTIVA/CERRADA/HISTÓRICA.
  • Registros diarios y fotos de progreso.
  • Análisis semanales completos.

SIN INTERVENCIÓN INVASIVA

El coach observa, no edita. La decisión deliberada es que la IA mantenga la coherencia del producto. Si un coach quiere cambiar algo, lo hace conversando con el alumno y dejando que la IA registre el cambio — así se preserva el historial y no se desencaja la lógica.

12 STACK TÉCNICO Y DEPENDENCIAS

FRONTEND

  • HTML/CSS/JS vanilla — un solo archivo index.html con todos los wrappers y la lógica.
  • Google Fonts — Bebas Neue, DM Sans, JetBrains Mono.
  • Sin framework — la simplicidad del producto no justifica React ni similares. Cero build step, cero dependencias en runtime.

BACKEND

  • Supabase — Postgres para datos relacionales, Storage para imágenes.
  • Edge Functions — endpoint para invocar Claude desde el navegador sin exponer la API key.
  • Row Level Security — cada usuario ve solo sus datos. El admin tiene política aparte.

IA

  • Anthropic Claude — modelo principal para el chat conversacional y la generación de planes.
  • Análisis multimodal — la IA recibe fotos junto con datos para los análisis semanales.
  • System prompts versionados — diferentes para el chat regular y para el análisis de cierre.

DEPLOY

  • Vercel — hosting estático del frontend con deploys automáticos desde main.
  • Branch-based workflow — todo va directo a main para iteración rápida.

FILOSOFÍA DEL CÓDIGO

El producto está construido con una decisión deliberada de simplicidad radical. No hay separación en componentes ni bundlers ni step de build. Todo vive en index.html porque eso mantiene el ciclo de iteración a segundos: editar, guardar, refrescar.

Esto tiene un costo (no escala como una app de producción a millones de usuarios sin refactorizar), pero el beneficio en velocidad de desarrollo en la fase actual del producto supera ampliamente esa deuda. Cuando crezca, se separa.
